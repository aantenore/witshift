# Wassette promotion gate

## Current status

WITShift targets [Microsoft Wassette](https://github.com/microsoft/wassette) as the capability-runtime
boundary. The opt-in positive-path smoke pins Wassette 0.4.0 and proves component loading, MCP tool
discovery, and invocation. It does not claim runtime-denial enforcement.

| Gate                                            | Status | Evidence label    |
| ----------------------------------------------- | ------ | ----------------- |
| Real Component generated by official tooling    | passed | build manifest    |
| Component function executed after jco transpile | passed | component-runtime |
| Candidate deny-by-default policy generated      | passed | artifact only     |
| Component loaded and invoked through Wassette   | passed | positive path     |
| Network/storage denial captured from Wassette   | open   | withheld          |
| MCP tool exposed through Wassette transport     | passed | positive path     |

The generated WIT world uses one direct export with a JSON string input and output per tool. This
preserves alpha tool semantics, avoids a runtime-specific nested interface adapter, and lets
capability runtimes expose each export as an MCP tool without pretending that arbitrary MCP or
Node.js host behavior was translated.

Run the opt-in gate with an evaluated binary:

```bash
WASSETTE_BIN=/absolute/path/to/wassette pnpm test:wassette
```

`WASSETTE_EXPECTED_VERSION` defaults to `0.4.0` and must be changed explicitly when evaluating a new
runtime. The command reports runtime/platform plus component and policy hashes. It never promotes
the verification report to `wassette-runtime` while denial evidence is absent.

## Promotion procedure

Pin an evaluated Wassette version, then capture all of the following in a replaceable execution
adapter and CI job:

1. runtime version and platform;
2. component and policy SHA-256 values;
3. successful component load;
4. tool discovery through the runtime's MCP surface;
5. one schema-valid invocation matching the original adapter;
6. a network target outside the allow list rejected by the runtime;
7. a filesystem path outside the allow list rejected by the runtime;
8. bounded diagnostics with secrets removed.

Only that adapter may declare `evidenceLevel: 'wassette-runtime'`, and each denial must identify
Wassette as its source with `runtimeEnforced: true`.

## Failure policy

- A load or invocation failure blocks the Wassette label, not static inspection evidence.
- A generated policy accepted syntactically but not enforced is a failure.
- An in-process allow-list evaluator can support a contract test but cannot close this gate.
- Runtime API drift belongs in the Wassette adapter; it must not add product-specific branches to the
  static analyzer or domain contracts.
