# Threat model

## Security claim

WITShift reduces migration ambiguity; it does not sandbox source code. The trusted runtime boundary
is expected to be a WebAssembly Component host such as Wassette. A generated policy is configuration,
not evidence that a host enforced it.

## Assets

- source and fixture confidentiality;
- integrity of emitted WIT, policy, component, and manifest;
- correctness of the original-to-component comparison;
- host filesystem, network, processes, and credentials;
- provenance of the component toolchain and downloaded build assets.

## Trust boundaries

1. `inspect` parses configured project source as data and does not import it.
2. `build` generates source only after the static gate, then invokes pinned external tooling without a
   shell and with a timeout.
3. `verify` imports configured adapter modules and executes them. Adapters are trusted code and can use
   Node.js privileges; do not run adapters from an untrusted repository on a sensitive host.
4. The component runtime is a separate boundary. jco execution proves component behavior, while only
   Wassette evidence can prove Wassette behavior.

## Threats and mitigations

| Threat                                      | Mitigation                                                                      | Residual risk                                               |
| ------------------------------------------- | ------------------------------------------------------------------------------- | ----------------------------------------------------------- |
| Dynamic source hides host behavior          | Reject dynamic import, eval, runtime resolution, process spawn, native addon    | Static analysis can still contain implementation defects    |
| Ambiguous registration or schema            | Require literal names/config and a documented schema subset                     | Unsupported valid servers require manual redesign           |
| Generated bytes are fake or wrong kind      | Invoke official tooling and validate the Component binary header                | Header validation is not semantic equivalence               |
| Differential input is mutated across ports  | Structured-clone and deep-freeze a separate input for each invocation           | Adapters still execute in one host process                  |
| Fixture or adapter hangs                    | Size/count bounds, abort signals, and per-invocation timeout                    | A malicious synchronous loop can still block Node.js        |
| Policy simulation is presented as isolation | Evidence records source and `runtimeEnforced`; report degrades to test-only     | Users can ignore labels outside WITShift                    |
| Cache poisoning                             | Cache key binds inputs/config/lock/toolchain/generated source; digest on use    | Local user with write access can delete the cache           |
| Upstream component bytes vary               | Compare independent runs and report the result; cache only stabilizes reuse     | Cross-host byte reproducibility is not established          |
| Archive extraction escapes its destination  | Development graph replaces the abandoned extractor with patched maintained fork | Consumers must use the documented secure peer setup         |
| Dependency or action drift                  | Exact package pins, lockfile, clean audit, immutable CI action commits          | New advisories can appear after release                     |
| Reports leak secrets                        | Source bodies and adapter stack traces are not emitted in stable CLI errors     | Fixture inputs/results can contain secrets; protect reports |

## Untrusted input guidance

- Inspecting untrusted source is lower risk than building or verifying it, but still run WITShift as a
  non-privileged user with a bounded workspace.
- Do not import untrusted verification adapters.
- Do not place credentials in fixtures, source literals, or report directories.
- Treat external componentization downloads as supply-chain inputs; use the locked setup and verify
  audits before release.
- Run final components under an actual capability runtime before production use.

## Non-goals

- proving arbitrary JavaScript safe;
- replacing a runtime sandbox;
- translating arbitrary Node.js APIs to WASI;
- guaranteeing semantic equivalence beyond supplied fixtures and schema checks;
- guaranteeing clean-room byte reproducibility while upstream componentization is nondeterministic.
